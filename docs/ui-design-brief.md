# Fastfood Kitchen UI 设计简报

发布日期：2026-04-08

## 1. 项目概述

Fastfood Kitchen 是一套面向连锁快餐与餐饮后厨场景的数字化管理系统，目标是将菜品标准、配方信息、制作 SOP、原料成本与门店执行流程统一到同一套平台中。

产品首版聚焦两个核心终端：

- Web 管理后台：服务总部、运营、店长或厨师长，用于配置和维护主数据
- 小程序端：服务门店后厨人员，用于现场查看菜品信息、配方与 SOP

本次 UI 设计目标不是做“展示型官网”，而是打造一套高效、稳定、易学、能支撑餐饮标准化落地的业务产品界面。

## 2. 设计目标

本轮 UI 设计需要同时满足以下目标：

- 让客户一眼理解这是一套“餐饮后厨标准化管理系统”
- 让总部用户可以高效维护门店、员工、原料、菜品、配方和 SOP
- 让后厨用户在高压、快节奏环境下快速找到并执行正确操作
- 让“成本”和“标准”成为页面中的高可见信息，而不是埋在数据里
- 为后续扩展更多模块时保留统一的产品视觉和交互基础

## 3. 业务定位

这是一个偏“运营管理 + 后厨执行”的业务系统，不是纯 ERP，也不是纯点餐产品。

它更像是餐饮品牌在内部使用的“标准化中枢”，主要连接三件事：

- 总部制定标准
- 门店执行标准
- 系统反馈成本与结构数据

## 4. 核心用户角色

### 4.1 管理员 / 总部运营

关注点：

- 统一管理门店与账号
- 维护原料与菜品资料
- 配置配方与 SOP
- 监控标准成本和建议售价

设计重点：

- 信息密度适中
- 导航清晰
- 表格、筛选、编辑效率高
- 重要指标可快速浏览

### 4.2 厨师长 / 门店管理者

关注点：

- 查看门店适用菜品
- 检查制作标准是否完整
- 确认人员执行是否一致

设计重点：

- 重点信息前置
- 详情结构清晰
- 支持快速比对和确认

### 4.3 厨师 / 一线后厨人员

关注点：

- 今天要做什么菜
- 每道菜怎么做
- 原料用量是多少
- 每一步在哪个工位完成
- 是否有注意事项

设计重点：

- 操作路径短
- 字大、层级清楚
- 高对比、低认知负担
- 适合厨房现场快速扫读

## 5. 核心业务场景

### 场景 A：总部建立标准菜品

运营或产品负责人在后台创建菜品，填写分类、工位、配方原料、损耗率、SOP 步骤和目标毛利，系统自动计算标准成本与建议售价。

UI 关键点：

- 菜品编辑页要成为核心工作台
- 配方、SOP、成本结果应在同一页形成闭环
- 自动计算结果要明显、可信、易复核

### 场景 B：门店培训与日常执行

厨师通过小程序查看菜品详情，快速了解配方、制作步骤、工位和耗时，按统一标准完成操作。

UI 关键点：

- 菜品详情要以“执行顺序”组织信息
- SOP 展示比后台字段更重要
- 避免复杂交互，优先清晰可读

### 场景 C：原料价格变化后的成本调整

管理端更新原料价格后，系统同步影响菜品成本和建议售价，帮助总部快速评估是否需要调整售价或配方。

UI 关键点：

- 原料页和菜品页需要建立关联感
- 成本变化应突出显示
- 利润空间、成本压力可视化要直观

### 场景 D：多门店标准同步

总部希望不同门店看到统一标准，但又保留门店、人员、角色的区分。

UI 关键点：

- 门店归属、角色信息、数据范围要清楚
- 页面中应体现“当前在哪个门店 / 哪个身份下”

## 6. 产品信息架构建议

### 6.1 Web 管理后台

建议一级导航：

- 登录
- 门店管理
- 用户管理
- 原料管理
- 菜品管理
- 菜品编辑 / 菜品详情

建议每个模块的定位：

- 门店管理：组织结构入口
- 用户管理：角色与权限入口
- 原料管理：成本输入入口
- 菜品管理：标准输出入口
- 菜品编辑：系统核心工作台

### 6.2 小程序端

建议页面结构：

- 登录页
- 菜品列表页
- 菜品详情页

后续可扩展页面：

- 我的
- 最近查看
- 收藏 / 常用菜品

## 7. 页面设计重点

### 7.1 登录页

目标：

- 快速进入系统
- 建立专业可信的第一印象

后台设计建议：

- 采用品牌化登录卡片
- 体现“后厨标准化管理”的定位信息
- 不要过度花哨，强调稳定和专业

小程序设计建议：

- 按移动端原生操作习惯设计
- 登录按钮突出
- 允许“继续当前会话”这类快捷入口

### 7.2 门店管理页

目标：

- 快速了解门店基础信息和状态

关键字段建议：

- 门店名称
- 门店编码
- 城市 / 区域
- 负责人
- 经营状态

设计建议：

- 表格视图为主
- 支持状态标签
- 适合后续增加筛选和门店切换

### 7.3 用户管理页

目标：

- 管理账号、角色和门店归属

关键字段建议：

- 用户名
- 显示名称
- 角色
- 所属门店
- 状态
- 联系方式

设计建议：

- 角色颜色或标签需有明显区分
- 用户编辑以抽屉或侧边表单体验更高效

### 7.4 原料管理页

目标：

- 成为成本数据的基础入口

关键字段建议：

- 原料名称
- 单位
- 单价 / 单位成本
- 规格
- 供应商
- 启用状态
- 更新时间

设计建议：

- 强调价格与规格信息
- 低库存、价格异常、停用状态可做视觉区分
- 适合后续加入趋势或变动提示

### 7.5 菜品管理页

目标：

- 快速查看菜品状态与经营价值

关键字段建议：

- 菜品名称
- 分类
- 工位
- 标准成本
- 建议售价
- 毛利率
- 状态
- 更新时间

设计建议：

- 不能只是普通 CRUD 列表
- 应强化“成本/毛利”这一业务亮点
- 可用颜色或轻量图形展示利润健康度

### 7.6 菜品编辑页

这是整个产品最关键的页面。

建议结构：

- 基础信息区
- 配方区
- SOP 区
- 成本结果区
- 保存与状态操作区

设计原则：

- 同页闭环，不要让用户频繁跳转
- 配方和 SOP 是主体，不应被压缩成附属信息
- 成本结果要固定在高可见区域
- 支持用户边编辑边理解成本变化

建议信息优先级：

1. 菜品基本身份信息
2. 配方原料明细
3. SOP 执行步骤
4. 标准成本与建议售价
5. 状态与保存反馈

### 7.7 小程序菜品列表页

目标：

- 让厨师快速找到对应菜品

设计建议：

- 卡片式列表优于复杂表格
- 每张卡片至少展示菜名、分类、工位
- 主按钮文案明确，例如“查看 SOP”
- 适合后续加入搜索、分类筛选、最近使用

### 7.8 小程序菜品详情页

这是小程序端的核心页面。

目标：

- 在最短时间内让后厨人员知道“这道菜怎么做”

建议内容顺序：

1. 菜品名称与分类
2. 工位与预计耗时
3. 原料与用量
4. SOP 步骤
5. 注意事项

设计建议：

- 按执行顺序展示，不要按数据库字段顺序展示
- SOP 步骤卡片要大、清楚、适合现场滑动浏览
- 每一步都应有明显编号
- 工位、耗时、关键提醒可以用标签强化

## 8. 视觉方向建议

### 8.1 整体气质

建议关键词：

- 专业
- 稳定
- 高效
- 标准化
- 餐饮现场感

避免方向：

- 过于互联网营销化
- 过于消费品电商化
- 过于 ERP 式陈旧灰表格

### 8.2 视觉风格建议

后台建议：

- 现代专业风格
- 深浅层级分明
- 模块边界清楚
- 强调结构和效率

小程序建议：

- 明亮直接
- 高可读
- 高对比
- 少装饰，重内容

### 8.3 色彩建议

建议以“食品安全 + 厨房秩序感”为方向建立色板：

- 主色可考虑深青、墨绿、炭蓝一类稳重色
- 强调色可使用暖橙或食材感颜色，用于提醒和关键动作
- 成本、风险、停用状态用红橙体系
- 正常、启用、可执行状态用绿色体系

不建议使用大面积梦幻色、荧光色或过于娱乐化配色。

### 8.4 字体与排版

建议重点：

- 标题和数值要有明显层级
- 毛利率、成本、售价等数字信息要易扫读
- 小程序端字号不宜过小
- SOP 内容排版优先可读性，不追求花样

## 9. 组件与交互建议

建议优先建立的基础组件：

- 顶部导航 / 侧边导航
- 数据表格
- 状态标签
- 搜索与筛选条
- 表单输入组件
- 动态列表编辑组件
- 步骤卡片
- 指标卡片
- 空状态
- 错误提示
- 成功反馈

关键交互原则：

- 保存要有清晰反馈
- 删除要有确认
- 异常状态要直接说明原因
- 空数据页不能留白，应给出下一步动作
- 页面加载时要让用户知道系统正在处理什么

## 10. 数据可视化建议

虽然首版不是 BI 系统，但以下内容值得轻量可视化：

- 标准成本
- 建议售价
- 目标毛利率
- 原料数量
- SOP 步骤数量
- 菜品状态

适合形式：

- 指标卡
- 标签
- 进度条
- 简单对比色块

不建议首版引入复杂大图表。

## 11. 品牌与文案建议

文案语气建议：

- 清晰
- 直接
- 专业
- 少口号，多动作

文案应尽量围绕操作与结果，例如：

- 保存菜品
- 更新原料价格
- 查看 SOP
- 重新计算成本
- 当前门店
- 当前角色

避免模糊文案，例如“提交信息”“处理数据”“系统设置”等泛化表述。

## 12. 首版 UI 优先级建议

如果时间有限，建议优先投入以下页面：

1. 后台登录页
2. 菜品管理页
3. 菜品编辑页
4. 原料管理页
5. 小程序菜品列表页
6. 小程序菜品详情页

原因：

- 这些页面最能体现产品价值
- 这些页面直接连接“标准化 + 成本 + 执行”三条主线
- 客户看演示时，也最容易通过这些页面理解产品能力

## 13. 输出物建议

建议本轮 UI 设计至少输出：

- 产品视觉基调板
- 后台信息架构图
- 小程序信息架构图
- 后台关键页面高保真稿
- 小程序关键页面高保真稿
- 基础组件规范
- 状态色和标签规范
- 菜品编辑页交互说明

## 14. 一句话设计结论

Fastfood Kitchen 的 UI 设计重点，不是把它做成“好看的后台”，而是把它做成一套能帮助餐饮品牌建立标准、控制成本、支撑门店执行的业务工具。
