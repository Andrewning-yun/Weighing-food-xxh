# 项目交付计划

发布日期：2026-04-08

## 1. 文档说明

本文档是当前项目的最新交付计划文件，用于指导团队按统一目标推进到“基本可交付”状态。

本文档聚焦：

- 交付目标
- 分阶段任务
- 岗位协作方式
- 验收标准
- 风险控制

本文档不覆盖首版之外的扩展功能，不包含 `apps/mobile` 的开发安排。

## 2. 交付目标

本轮交付目标为完成一个可试运行的 MVP 版本，必须满足以下条件：

- 后端服务可安装、可构建、可启动
- 管理后台可完成核心主数据维护
- 小程序端可完成菜品查看与 SOP 查看
- 登录鉴权可用
- 原料、菜品、用户、门店四类核心数据可维护
- 成本计算可返回标准成本与建议售价
- 至少具备最小可用测试集

## 3. 本轮交付范围

### 纳入交付

- `services/api`
- `apps/web-admin`
- `apps/miniapp`
- `packages/domain`
- `docs`

### 暂不纳入交付

- `apps/mobile`
- 高级报表
- 审批流
- 批量导入导出
- 对象存储上传
- 多组织复杂权限体系

## 4. 当前问题摘要

当前版本距离交付仍有明显差距，主要问题如下：

- 根目录构建链路未打通
- 包管理器策略未统一
- 后端模块存在缺失引用
- 管理后台仍为空
- 小程序只有请求层，没有页面能力
- 测试体系仍未建立
- 项目尚未形成稳定验收机制

因此，本计划的首要任务不是“继续铺功能”，而是“先形成最小可运行闭环”。

## 5. 分阶段交付安排

### 阶段 0：工程基线修复

目标：

- 修复当前无法构建、无法稳定启动的问题

必须完成：

- 统一包管理器
- 修复根目录 `build`、`dev`、`lint` 命令
- 更新 `README.md`
- 补齐 `.env.example`
- 明确 Node 版本
- 保证 `services/api` 至少可以独立构建

输出物：

- 可执行的安装说明
- 可执行的启动说明
- 可执行的构建说明

完成标准：

- 新成员可根据文档安装依赖
- 根目录命令可运行
- 后端服务能启动

### 阶段 1：后端 MVP 闭环

目标：

- 提供给前端联调的稳定 API

必须完成模块：

- `auth`
- `store`
- `user`
- `ingredient`
- `dish`
- `costing`

必须完成能力：

- 登录
- JWT 鉴权
- 角色控制
- 门店 CRUD
- 用户 CRUD
- 原料 CRUD
- 菜品 CRUD
- 成本计算

必须补齐：

- DTO
- Service
- Module
- Guard
- Decorator
- Migration
- Seed

完成标准：

- 核心接口可通过 Swagger 或 Postman 验证
- API 可供前端稳定联调
- 至少一套演示数据可用

### 阶段 2：管理后台交付

目标：

- 实现主数据维护后台

必须完成页面：

- 登录页
- 门店管理页
- 用户管理页
- 原料管理页
- 菜品管理页
- 菜品编辑页

必须完成能力：

- 登录态校验
- 列表展示
- 新增
- 编辑
- 删除或停用
- 表单校验
- 错误提示

完成标准：

- 管理员可独立维护基础数据
- 菜品编辑页可维护配方与 SOP
- 菜品详情能展示成本结果

### 阶段 3：小程序交付

目标：

- 实现厨师查看型端能力

必须完成页面：

- 登录页
- 菜品列表页
- 菜品详情页
- 我的页面

必须完成能力：

- 登录态缓存
- 菜品列表展示
- 菜品详情展示
- 配方与 SOP 查看
- 错误态提示

完成标准：

- 厨师可以登录
- 能查看菜品详情
- 能查看配方和 SOP

### 阶段 4：测试与发布加固

目标：

- 达到基本可交付水准

必须完成：

- 后端核心接口测试
- 管理后台冒烟测试
- 小程序冒烟测试
- 回归清单
- 部署文档
- 演示账号说明

完成标准：

- 所有 P0 问题关闭
- 核心路径全部通过验收
- 演示环境可运行

## 6. 岗位执行要求

### 项目负责人

负责：

- 冻结范围
- 控制节奏
- 审批延期项
- 组织验收

要求：

- 不允许范围反复变化
- 不允许未验收就进入下一阶段

### 后端工程师

负责：

- `services/api`
- 相关领域模型

要求：

- 先修缺失模块和构建错误
- 再补完整业务闭环
- 每个接口都要能联调，不允许只写空壳

### 管理后台前端工程师

负责：

- `apps/web-admin`

要求：

- 以 CRUD 为核心
- 不优先追求复杂 UI
- 页面必须能联调可用

### 小程序工程师

负责：

- `apps/miniapp`

要求：

- 先做查看型功能
- 不提前做复杂编辑
- 必须保证真实设备或开发工具中可访问

### 测试工程师

负责：

- 核心流程测试
- 回归测试
- 缺陷跟踪

要求：

- 必须给出明确可交付结论
- 不允许只做口头验证

### DevOps / 发布支持

负责：

- 构建链路
- 环境变量
- 启动流程
- 发布文档

要求：

- 保证新环境可复现
- 保证构建与启动命令稳定

## 7. 交付验收标准

本轮交付必须同时满足以下条件：

- 根目录安装命令可执行
- 根目录构建命令可执行
- 后端服务可启动
- 登录接口可用
- 原料、菜品、用户、门店四类管理功能可用
- 成本计算结果可返回
- 管理后台可维护主数据
- 小程序可查看菜品详情和 SOP
- 最小测试集已补齐
- 部署文档完整

任何一项不满足，都不能定义为“已完成交付”。

## 8. 风险与控制策略

### 风险 1：继续扩展范围

控制策略：

- 所有新增功能必须经过项目负责人确认
- 当前阶段禁止新增 `apps/mobile` 工作

### 风险 2：只补文件，不验证运行

控制策略：

- 每项任务完成后必须附带运行结果
- 不能以“代码写了”作为完成依据

### 风险 3：前后端接口反复变动

控制策略：

- 先冻结接口结构
- 变更字段前必须同步

### 风险 4：缺少测试导致返工

控制策略：

- 核心路径优先补测试
- 每个阶段结束必须做回归

## 9. 建议执行顺序

推荐团队按以下顺序推进：

1. 修工程基线
2. 修后端缺失模块
3. 打通后端核心业务
4. 搭建管理后台并完成 CRUD
5. 搭建小程序查看页
6. 做联调与测试
7. 做发布准备

## 10. 交付结果导向要求

从当前版本开始，所有工作汇报必须改成结果导向，不再使用“目录建好了”“文件补了”“准备做完了”作为结论。

统一使用以下口径汇报：

- 是否可安装
- 是否可构建
- 是否可启动
- 是否可联调
- 是否可演示
- 是否可回归

只有这些结果成立，才算进入真实交付阶段。
