# 项目审计与 MVP 交付计划

## 1. 报告目的

本报告用于回答两个问题：

1. 当前项目做到什么程度了。
2. 团队接下来如何把它推进到“可以基本交付”的水准。

这里的“基本交付”定义为：

- 能完成登录鉴权。
- 能管理门店、用户、原料、菜品、配方、SOP。
- 能进行基础成本计算和建议售价输出。
- 管理后台可完成配置和维护。
- 小程序端可供厨师查看菜品配方与 SOP。
- 系统可以在一套稳定的开发/测试/预发布流程下运行。

不纳入本轮基本交付范围：

- React Native `apps/mobile`
- 复杂 BI 分析
- 高级运营报表
- 多品牌复杂结算
- 完整工作流审批

## 2. 当前状态结论

### 2.1 总体判断

当前仓库处于“骨架已搭建、业务尚未落地”的早期阶段，整体完成度约为 10%。

更准确地说：

- 工程骨架完成度约 25%
- 可交付产品完成度约 5% 到 10%
- 当前阶段仍属于 README 中的 Phase 1，而非可试运行阶段

### 2.2 已完成内容

- 已建立 monorepo 基本目录结构。
- 根目录已有 workspace 配置。
- 后端已有 NestJS 启动入口。
- 后端已定义部分核心实体：
  - `Store`
  - `User`
  - `Ingredient`
  - `Dish`
- 已有共享领域类型 `packages/domain`。
- 小程序端已有基础请求封装。

### 2.3 未完成内容

- `apps/web-admin` 为空目录。
- `apps/mobile` 为空目录。
- `packages/api-client`、`packages/config`、`packages/tooling`、`packages/ui` 为空目录。
- `docs` 原先为空目录。
- 小程序端没有页面、组件、状态管理、业务 API 实现。
- 后端没有完整模块实现，缺少：
  - module
  - controller
  - service
  - dto
  - migration
  - seed
  - auth 流程闭环
  - costing 核心逻辑
- 无自动化测试。
- 无初始化数据方案。
- 无部署说明。
- 当前仓库无提交历史，说明项目仍未进入稳定协作节奏。

## 3. 关键问题诊断

### 3.1 真实问题不是“做得慢”，而是“范围先铺开了，交付路径没收敛”

当前仓库同时铺了：

- 小程序
- 管理后台
- 移动端
- 后端
- 多个共享包

但真正有代码的只有：

- 后端入口和少量实体
- 小程序请求层
- 共享类型定义

这会导致团队产生一种“架子已经很完整”的错觉，但实际还没有形成最小闭环。

### 3.2 基础技术路径没有闭环

目前至少存在以下工程风险：

- README 与实际环境不一致，文档要求 `yarn`，但当前环境只确认有 `node` 和 `npm`。
- 根仓库存在 `package-lock.json`，说明包管理策略未统一。
- 后端入口导入了多个模块，但模块文件并未形成完整实现。
- 没有测试，意味着每次迭代都会靠人工试错。

### 3.3 没有 MVP 切割，导致交付目标模糊

如果团队继续按当前方式并行展开，最可能的结果是：

- 每个端都有一点代码
- 但没有一个端能完整跑通
- 到后期只能靠大量返工收敛

## 4. 建议的 MVP 范围

必须先收缩范围，聚焦一个能落地的版本。

### 4.1 推荐 MVP 目标

本轮只做两端一后端：

- `services/api`
- `apps/web-admin`
- `apps/miniapp`

### 4.2 MVP 必做功能

#### A. 账号与权限

- 用户登录
- JWT 鉴权
- 角色区分：
  - 管理员
  - 厨师长
  - 厨师
- 基于门店的数据隔离

#### B. 门店与用户管理

- 创建门店
- 创建用户
- 用户绑定门店
- 用户角色分配

#### C. 原料管理

- 原料列表
- 新增/编辑/停用原料
- 单位与价格维护
- 单位成本字段计算或录入规则

#### D. 菜品与配方管理

- 菜品基本信息
- 菜品分类
- 工位信息
- 配方原料列表
- 原料用量、单位、损耗率

#### E. SOP 管理

- 步骤列表
- 每一步标题、描述、耗时、工位
- 图片字段先保留为 URL，不做复杂上传系统

#### F. 成本计算

- 依据原料价格、用量、损耗率计算标准成本
- 根据目标毛利率输出建议售价
- 菜品详情页展示成本拆解

#### G. 小程序厨师端

- 登录
- 菜品列表
- 菜品详情
- 查看配方
- 查看 SOP

#### H. 管理后台

- 登录页
- 原料管理页
- 菜品管理页
- 用户管理页
- 门店管理页

## 5. 明确不做的内容

为了保证交付，以下内容必须延期：

- `apps/mobile` 原生端
- 复杂图表和多维经营分析
- 高级搜索
- 批量导入导出
- 图片上传到对象存储
- 审批流
- 多组织多品牌复杂权限矩阵
- 消息通知系统

## 6. 建议的目标架构

### 6.1 后端

采用 NestJS + PostgreSQL，保持当前方向，但补齐标准分层：

- `module`
- `controller`
- `service`
- `entity`
- `dto`
- `guard`
- `decorator`
- `migration`
- `seed`

推荐最先落地的模块：

1. `auth`
2. `user`
3. `store`
4. `ingredient`
5. `dish`
6. `costing`

### 6.2 管理后台

推荐采用 Next.js，仅做后台管理使用，不做复杂设计系统，先以 CRUD 为目标。

页面建议：

- `/login`
- `/stores`
- `/users`
- `/ingredients`
- `/dishes`
- `/dishes/[id]`

### 6.3 小程序

继续使用 Taro，但只做“查看型”能力，不在首版承担复杂编辑。

页面建议：

- 登录页
- 菜品列表页
- 菜品详情页
- 我的信息页

## 7. 分阶段交付路线

建议按 4 个阶段推进，每个阶段结束都必须有可验收结果。

### 阶段 0：项目收敛与清障，1 到 2 天

目标：

- 团队统一交付范围
- 工程环境统一
- 跑通最基础开发流程

任务：

- 确定本轮不做 `apps/mobile`
- 统一包管理器，只保留一种
- 修正 README
- 建立 `.env.example`
- 约定 Node 版本
- 约定 lint / build / test 命令
- 初始化 Git 仓库与分支策略

完成标准：

- 新成员可按文档在本地启动项目
- 根目录命令可以稳定执行

### 阶段 1：后端 MVP 闭环，4 到 6 天

目标：

- 提供可供前端联调的稳定 API

任务：

- 实现 `auth` 模块
  - 登录
  - JWT 签发
  - 路由守卫
- 实现 `store` 模块 CRUD
- 实现 `user` 模块 CRUD
- 实现 `ingredient` 模块 CRUD
- 实现 `dish` 模块 CRUD
- 实现 `costing` 服务
  - 菜品成本计算
  - 建议售价输出
- 增加 DTO 校验
- 增加数据库 migration
- 增加 seed 数据

完成标准：

- Postman 或 Swagger 可验证全部核心接口
- 一套 seed 数据可跑出演示环境

### 阶段 2：管理后台可用，4 到 5 天

目标：

- 让运营/管理员可以完成主数据配置

任务：

- 实现登录页
- 实现用户列表与编辑
- 实现门店列表与编辑
- 实现原料列表与编辑
- 实现菜品列表与编辑
- 菜品详情页支持编辑配方与 SOP
- 接入后端鉴权
- 表单错误处理

完成标准：

- 管理员能独立创建门店、用户、原料、菜品
- 菜品保存后可看到成本结果

### 阶段 3：小程序厨师端可用，3 到 4 天

目标：

- 让厨师能在现场查看菜品操作信息

任务：

- 实现登录
- 实现菜品列表
- 实现菜品详情
- 展示配方、工位、耗时、SOP
- 增加基础缓存与错误提示

完成标准：

- 厨师账号可登录并查看本门店菜品信息

### 阶段 4：交付加固，2 到 3 天

目标：

- 达到“可以交付试用”的最低质量线

任务：

- 冒烟测试
- 修复高优先级 bug
- 增加关键接口测试
- 增加关键页面回归测试清单
- 输出部署说明
- 输出演示账号与演示数据

完成标准：

- 管理后台和小程序都能完成核心路径演示
- 后端部署步骤清晰

## 9. 优先级排序

必须按下面顺序做，不建议打乱：

1. 统一工程环境
2. 做通后端鉴权
3. 做通原料与菜品 CRUD
4. 做通成本计算
5. 做通后台管理
6. 做通小程序查看
7. 最后再补测试、部署、文档细节

原因很简单：

- 没有后端闭环，前端无法稳定联调
- 没有原料和菜品，成本计算没有输入
- 没有后台配置能力，小程序也没有真实数据可看

## 10. 必须补齐的数据设计点

现有实体可以继续用，但建议尽快补齐以下字段或规则：

### Store

- 是否启用
- 联系人
- 联系电话

### User

- 状态
- 最后登录时间
- 密码加密策略

### Ingredient

- 规格说明
- 供应商
- 是否启用
- 生效时间

### Dish

- 归属门店或品牌范围
- 菜品编码
- 版本号
- 成本更新时间

### 成本规则

必须尽快明确：

- 单位换算规则
- 损耗率默认值
- 毛利率算法
- 建议售价取整规则

如果这些规则不明确，后续前后端都容易返工。

- 后端可以通过统一命令启动

- 部署步骤可复现

## 14. 项目负责人需要立刻拍板的事项

以下事项如果不拍板，团队会持续空转：

1. 首版是否明确放弃 `apps/mobile`
2. 包管理器最终使用 `npm` 还是 `yarn`
3. 后端接口是否由 NestJS 单体先完成，不拆服务
4. 首版是否只支持单品牌/单组织简化模型
5. 成本计算规则是否先采用固定公式
6. 图片上传是否延期，仅保留 URL 字段

建议全部选择“先简化”，保证先交付。

## 15. 最终建议

这个项目并不是不可救，也不是推倒重来型问题。

真正需要做的是：

- 停止继续铺新目录
- 收缩到一个明确 MVP
- 先做通后端和后台
- 用小程序承接查看场景
- 把移动端和高级能力整体延后

只要团队能在未来 2 到 3 周内严格按本报告收敛范围并执行，项目是有机会达到“可基本交付、可试运行”的。

反过来说，如果继续多端并行铺开、没有验收节点、没有测试和环境统一，这个仓库会很快进入高返工状态。
