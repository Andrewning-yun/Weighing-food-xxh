# 岗位分工与功能实现说明

## 1. 文档目的

本文档用于把当前项目的整改目标拆解到具体岗位，明确每个角色需要负责的功能、输出物、协作边界和完成标准。

适用范围：

- `services/api`
- `apps/web-admin`
- `apps/miniapp`
- `packages/domain`
- `docs`

本文档默认项目目标为“达到基本交付水准的 MVP 版本”，不覆盖 `apps/mobile` 首版开发。

## 2. 当前项目的目标系统

目标系统是一个面向餐饮后厨的菜品与配方管理系统，首版必须完成以下闭环：

- 用户登录与权限控制
- 门店管理
- 用户管理
- 原料管理
- 菜品与配方管理
- SOP 管理
- 成本计算
- 管理后台维护数据
- 小程序端查看菜品、配方和 SOP

## 3. 岗位定义与责任边界

建议项目团队至少包含以下岗位：

1. 项目负责人 / 技术负责人
2. 后端工程师
3. 管理后台前端工程师
4. 小程序工程师
5. 测试工程师
6. DevOps / 发布支持

如果团队人数不足，可以一人兼任多个岗位，但责任边界不能缺失。

## 4. 项目负责人 / 技术负责人

### 4.1 主要职责

项目负责人不是“写所有代码的人”，而是“保证系统能按顺序交付的人”。

必须承担以下工作：

- 冻结 MVP 范围
- 定义技术方案和目录边界
- 决定包管理器、环境规范、分支策略
- 确认 API 规范与数据模型
- 拍板延期项和不做项
- 管理阶段验收
- 控制返工风险

### 4.2 必须输出的内容

- MVP 功能清单
- 环境规范文档
- API 路由规范
- 数据模型说明
- 每阶段验收标准
- 缺陷优先级规则

### 4.3 当前阶段必须推动完成的事项

- 决定根仓库只使用一种包管理器
- 修复根目录构建命令不可执行的问题
- 明确首版不开发 `apps/mobile`
- 确认鉴权方式为 JWT 单体方案
- 确认首版成本计算规则
- 统一数据库初始化方案

### 4.4 完成标准

- 团队成员知道自己负责哪个目录
- 开发、联调、测试、发布的入口命令已统一
- 每个阶段都有验收结果，不再以“写了几个文件”作为完成依据

## 5. 后端工程师

### 5.1 负责目录

- `services/api`
- 必要时同步更新 `packages/domain`

### 5.2 核心职责

后端工程师必须把当前 NestJS 项目从“骨架”补成“可运行服务”。

必须完成以下模块：

1. `auth`
2. `store`
3. `user`
4. `ingredient`
5. `dish`
6. `costing`

### 5.3 必须实现的功能

#### A. 认证与权限

必须实现：

- 登录接口
- JWT 签发
- 鉴权 Guard
- 角色装饰器
- 基于角色的路由保护

建议接口：

- `POST /auth/login`
- `GET /auth/me`

#### B. 门店管理

必须实现：

- 创建门店
- 门店列表
- 门店详情
- 更新门店
- 门店启停状态

建议接口：

- `POST /stores`
- `GET /stores`
- `GET /stores/:id`
- `PATCH /stores/:id`

#### C. 用户管理

必须实现：

- 创建用户
- 查询用户
- 修改用户
- 重置密码
- 分配角色
- 绑定门店

建议接口：

- `POST /users`
- `GET /users`
- `GET /users/:id`
- `PATCH /users/:id`
- `PATCH /users/:id/password`

#### D. 原料管理

必须实现：

- 新增原料
- 查询原料列表
- 查询单个原料
- 更新原料
- 删除或停用原料

必须补齐字段：

- `name`
- `unit`
- `price`
- `costPerUnit`
- `supplier`
- `spec`
- `isActive`

建议接口：

- `POST /ingredients`
- `GET /ingredients`
- `GET /ingredients/:id`
- `PATCH /ingredients/:id`
- `DELETE /ingredients/:id`

#### E. 菜品与配方管理

必须实现：

- 创建菜品
- 查询菜品列表
- 查询菜品详情
- 更新菜品
- 上下架菜品

菜品结构必须覆盖：

- 菜品名称
- 菜品分类
- 工位
- 配方原料列表
- 原料用量
- 单位
- 损耗率
- SOP 步骤
- 预计毛利率
- 标准成本
- 建议售价

建议接口：

- `POST /dishes`
- `GET /dishes`
- `GET /dishes/:id`
- `PATCH /dishes/:id`
- `PATCH /dishes/:id/status`

#### F. 成本计算服务

必须实现：

- 按菜品计算总成本
- 按原料拆解成本明细
- 根据毛利率返回建议售价
- 原料价格变动后触发菜品成本重算

建议服务能力：

- `calculateDishCost(dishId)`
- `recalculateAllDishes()`
- `estimateSuggestedPrice(totalCost, grossMargin)`

### 5.4 工程职责

后端工程师不能只写 Controller，必须同时完成：

- DTO 定义
- 参数校验
- Entity 修正
- Module 装配
- Service 实现
- 数据库 migration
- seed 数据
- Swagger 或 Postman 文档
- `.env.example` 所需变量说明

### 5.5 当前必须修复的问题

根据本次审计，后端当前必须优先处理：

- `main.ts` 中引用了不存在的模块
- `ingredient` 模块引用了不存在的 DTO 和鉴权文件
- `CostingService` 尚未实现
- 全局前缀与 Controller 路径重复，需修正 `/api/api/...` 风险
- 当前无法完成稳定构建

### 5.6 完成标准

后端岗位完成的标志不是“文件存在”，而是：

- `services/api` 可以独立启动
- 登录接口可用
- 原料、菜品、用户、门店接口可联调
- 成本计算返回正确结构
- 至少具备最小测试集

## 10. 协作规则

为了避免继续出现“只有局部文件增加，但整体不可交付”的情况，团队必须按以下规则协作：

### 10.1 分支与提交规则

- 每个岗位按模块提交，不要跨目录随意扩散
- 每项功能必须包含最小可验证结果
- 提交描述必须说明功能和影响范围

### 10.2 联调规则

- 后端先给出接口契约
- 前端按契约联调
- 改字段前必须同步通知相关岗位
- 不允许在前后端各自假设不同数据结构

### 10.3 完成定义

任务完成必须同时满足：

- 代码已写完
- 本地可运行
- 已自测
- 已补必要文档
- 已通过对应验收

只改代码、不验证运行，不算完成。

## 11. 本周按岗位的执行顺序

### 项目负责人

- 冻结 MVP 范围
- 统一包管理器和验收口径
- 决定延期项

### 后端工程师

- 先修复模块缺失和构建错误
- 再做 auth / user / store
- 再做 ingredient / dish / costing

### 管理后台前端工程师

- 先初始化项目和登录页
- 再做原料、菜品、用户、门店页面

### 小程序工程师

- 先补页面结构和路由
- 再接登录
- 再接菜品列表和详情

### 测试工程师

- 先写核心路径测试用例
- 后跟随联调节奏做回归

### DevOps / 发布支持

- 先修安装与构建
- 再补环境文档和启动文档

## 12. 岗位验收口径

### 项目负责人验收口径

- 项目目标是否清晰
- 范围是否收敛
- 团队是否按计划推进

### 后端工程师验收口径

- API 是否真实可用
- 服务是否可启动
- 核心模块是否齐全

### 管理后台前端工程师验收口径

- 页面是否可用
- CRUD 是否完整
- 是否完成联调

### 小程序工程师验收口径

- 页面是否真实可访问
- 菜品详情是否可展示
- 登录态是否稳定

### 测试工程师验收口径

- 是否覆盖核心路径
- 是否能明确指出阻塞问题

### DevOps / 发布支持验收口径

- 新环境是否可复现
- 构建和启动是否稳定

## 13. 最终要求

从本周开始，团队所有岗位都应停止以“代码量”“文件数”“目录结构”作为进度证明，统一改为以下结果导向：

- 能不能安装
- 能不能构建
- 能不能启动
- 能不能联调
- 能不能演示
- 能不能回归

只有达到这些结果，项目才算真正向“基本交付”推进。
